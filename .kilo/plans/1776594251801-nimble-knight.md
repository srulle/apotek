# Rencana Desain Tabel Stok untuk Sistem Apotek

## Analisis Database Saat Ini

Berdasarkan migrasi yang ada, sistem memiliki:

- **Tabel obat**: Informasi obat dengan unit, harga
- **Tabel pembelian**: Header pembelian dengan supplier, tanggal, user
- **Tabel pembelian_detail**: Baris item pembelian dengan nomor batch, tanggal expired, jumlah

Observasi kunci:

- Pelacakan batch sudah ada (`nomor_batch`, `tanggal_expired`)
- Sistem penjualan belum diimplementasi
- Belum ada tabel pelacakan stok saat ini

## Desain Tabel Stok yang Direkomendasikan

### Tabel: stok

```sql
CREATE TABLE stok (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    obat_id BIGINT UNSIGNED NOT NULL,
    nomor_batch VARCHAR(50) NOT NULL,
    tanggal_expired DATE NOT NULL,
    stok INT UNSIGNED NOT NULL DEFAULT 0,
    created_at TIMESTAMP NULL,
    updated_at TIMESTAMP NULL,

    FOREIGN KEY (obat_id) REFERENCES obat(id) ON UPDATE CASCADE ON DELETE RESTRICT,
    INDEX idx_obat_batch (obat_id, nomor_batch),
    INDEX idx_tanggal_expired (tanggal_expired),
    INDEX idx_stok (stok),
    UNIQUE KEY unique_obat_batch (obat_id, nomor_batch)
);
```

### Alasan

- **Pelacakan per-batch**: Karena sistem melacak batch dengan tanggal expired, stok harus dikelola per batch untuk kontrol inventori yang tepat
- **Siap FIFO**: Index `tanggal_expired` memungkinkan manajemen inventori First In First Out untuk penjualan masa depan
- **Desain sederhana**: Tidak ada field lokasi atau alert sesuai permintaan
- **Kendala**: Mencegah entri batch duplikat per obat

## Strategi Manajemen Stok

### Pendekatan Implementasi Laravel

Gunakan **Model Observers** atau **Model Events** untuk mengelola stok secara otomatis:

1. **Event Pembelian** (Model PembelianDetail):
    - `created`: Tambah jumlah stok
    - `updated`: Sesuaikan stok jika jumlah berubah
    - `deleted`: Kurangi jumlah stok

2. **Event Penjualan Masa Depan** (Model PenjualanDetail - saat dibuat):
    - `created`: Kurangi stok menggunakan logika FIFO (batch tertua dulu)
    - `updated`: Sesuaikan jumlah stok
    - `deleted`: Kembalikan jumlah stok

### Logika Update Stok

- **Tambah stok**: `stok += jumlah_beli` saat detail pembelian dibuat
- **Kurangi stok**: `stok -= jumlah_jual` saat detail penjualan dibuat (pilih batch tertua dulu)
- **Seleksi batch**: Untuk penjualan, gunakan FIFO - pilih batch dengan tanggal expired terawal dulu

## Hubungan Antar Tabel

### Koneksi Tabel Stok dengan Tabel Lain

1. **Stok ↔ Obat**:
    - `stok.obat_id` → `obat.id` (Foreign Key)
    - Relasi: One-to-Many (satu obat bisa punya banyak stok batch)

2. **Stok ↔ PembelianDetail**:
    - Tidak ada foreign key langsung
    - Koneksi melalui: `stok.obat_id = pembelian_detail.obat_id` AND `stok.nomor_batch = pembelian_detail.nomor_batch`
    - `stok.tanggal_expired` harus sama dengan `pembelian_detail.tanggal_expired`

3. **Stok ↔ PenjualanDetail** (masa depan):
    - Koneksi melalui obat_id dan nomor_batch
    - Logic FIFO akan memilih batch dengan tanggal_expired terlama untuk penjualan

### Field yang Menghubungkan

- **obat_id**: Menghubungkan Stok dengan Obat dan PembelianDetail dengan Obat
- **nomor_batch**: Menghubungkan Stok dengan PembelianDetail (batch tracking)
- **tanggal_expired**: Menghubungkan Stok dengan PembelianDetail (expiry tracking)

### Catatan Implementasi

- Tidak perlu menambah field baru pada tabel existing
- Koneksi sudah tersedia melalui field yang ada
- Observer PembelianDetail akan mencari/update Stok berdasarkan kombinasi (obat_id, nomor_batch)
